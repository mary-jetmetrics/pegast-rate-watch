# Курс Пегаса

Каждое утро (около 07:40 по Перми) присылает в Telegram курс евро туроператора
Pegas Touristik и копит историю в [`history.csv`](history.csv).

Время приблизительное намеренно: расписания в GitHub Actions не имеют гарантий,
запуск может задержаться на час-другой, а при пиковой нагрузке и вовсе пропасть.
Данные от этого не страдают — пропущенный день добирается задним числом.

Нужно, чтобы поймать выгодный момент для доплаты остатка за тур на Фукуок
(бронь от 10.07.2026, зафиксирована в евро — значит важен только курс, по которому
пересчитывается каждый платёж).

## Откуда данные

Страница `agency.pegast.ru/ExchangeRates` рисует таблицу через JS, но под ней лежит
открытый эндпоинт, отдающий курс на **любую** дату:

```
POST https://agency.pegast.ru/ExchangeRates/GetExchangeRates
Content-Type: application/json

{"date": "2026-06-15"}
```

Отвечает `{"Rates":[{"SourceCurrency":"EUR","Rate":88.5800, ...}]}`. Авторизация не нужна.

Ответ — не строгий JSON: даты приходят литералом `new Date(1783641600000)`, поэтому
перед разбором его чиним регуляркой.

## Когда Пегас ставит курс

По будням после 17:30 МСК — на следующий день. Пятничный курс держится всю
субботу, воскресенье и понедельник (10 июля — 92,44, а 11–13 июля — 93,58).
Слать одно и то же сообщение три дня подряд смысла нет, поэтому запуск только
по будням. Дырок в истории при этом не возникает: каждый запуск добирает все
даты, которых нет в `history.csv`.

Для нас важно, что к утру курс на сегодня уже готов и меняться не должен. Гнаться
за минутой публикации не надо, надо лишь успеть до начала рабочего дня турагента —
платим через неё, а она Пегасу.

## Почему история перечитывается, а не пишется один раз

В волатильные дни оператор оставляет за собой право переставить курс несколько раз
за день, в том числе после того, как мы его записали. Поэтому каждый запуск
перечитывает последние `RECHECK_DAYS` дней поверх сохранённого и, если значение
разошлось, чинит историю и говорит об этом в сообщении.

Само по себе это не выяснить у API: **он не умеет отвечать «курса на эту дату ещё
нет»**. На неустановленную дату он молча отдаёт последний известный курс — на год
вперёд вернёт сегодняшний. Поэтому спрашивать будущие даты бессмысленно, а `None`
из `fetch_rate` означает сбой API, а не отсутствие курса.

## Настройка

Секреты репозитория (Settings → Secrets and variables → Actions):

| Секрет | Что это |
|---|---|
| `TG_BOT_TOKEN` | токен бота от [@BotFather](https://t.me/BotFather) |
| `TG_CHAT_ID` | ID группы, куда слать (отрицательное число) |

Константы брони — вверху `rates.py`: `BOOKING_DATE`, `BOOKING_RATE`, `TOUR_TOTAL_EUR`.

## Как учитываются платежи

Тур стоит 3 423,60 €. Каждый платёж пересчитывается в евро по курсу того дня,
когда он прошёл, и уменьшает остаток. Список — в `PAYMENTS`:

```python
PAYMENTS = [
    (dt.date(2026, 7, 10), 65_000, 92.44, 703.15),  # дата, ₽, курс, зачтено €
]
```

Зачтённые евро берутся **из учёта туроператора**, а не считаются заново:
он округляет вниз (65 000 / 92,44 = 703,159 €, зачёл 703,15 €), и остаток
должен сходиться с его цифрами.

**Внесли доплату — допишите строку в `PAYMENTS`.** Остаток и итоговая сумма
тура пересчитаются сами.

## Локальный запуск

```sh
python3 rates.py --dry-run   # показать сообщение, не отправлять
python3 rates.py             # отправить (нужны TG_BOT_TOKEN и TG_CHAT_ID в окружении)
```

Зависимостей нет, только стандартная библиотека.
